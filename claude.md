# CLAUDE.md

## 项目概述

百得思维官网（ibestservices.com）创新业务部改版项目。
本次任务：新增"RPO及猎头服务"二级页面，并在首页导航栏增加对应入口。
项目地址：https://www.ibestservices.com/
本页面为纯静态展示页，所有文案和数据直接写在代码中，无需调用任何后端接口。
本页面的设计风格与交互方式应该与项目的其他模块页面一致，首次开发时需要访问类似模块页面查看后统一：https://www.ibestservices.com/business_types/project_service/SJservice ，https://www.ibestservices.com/business_types/technical_service/

**项目需求文档**：详见根目录 `创新业务部改版需求.pdf`

## 项目初始化

本项目从零开始搭建。首次启动时请帮我完成以下初始化工作：

1. 使用 Vite 创建 React + TypeScript 项目
2. 安装并配置 Tailwind CSS
3. 安装并初始化 shadcn/ui
4. 安装 lucide-react
5. 创建以下目录结构：

```
src/
├── pages/          # 页面文件
│   └── rpo/        # RPO及猎头服务页面
│       └── index.tsx
├── components/     # 可复用组件
│   ├── layout/     # 布局相关组件（导航栏、页脚等）
│   │   ├── Navbar.tsx
│   │   └── Footer.tsx
│   └── rpo/        # RPO页面专用组件
│       ├── HeroBanner.tsx          # 顶部大图区域
│       ├── ServiceCategories.tsx   # 业务分类（中高端人才寻访、RPO招聘流程外包、招聘团队驻场服务）
│       ├── ServiceProcess.tsx      # 服务流程（14步，分四阶段）
│       ├── CaseStudy.tsx           # 合作案例
│       ├── ServiceScope.tsx        # 服务范围
│       ├── ProjectRequirements.tsx # 项目需求
│       ├── ExecutionData.tsx       # 执行数据
│       ├── ProjectHighlights.tsx   # 项目亮点
│       ├── Glory.tsx               # 荣耀
│       └── Commitment.tsx          # 我们承诺
├── types/          # TypeScript 类型定义
│   └── rpo.ts
├── App.tsx         # 根组件，包含路由
└── main.tsx        # 入口文件
```

## 目录结构说明

| 目录/文件 | 作用 |
|-----------|------|
| `src/pages/` | 放置各个页面，一个页面一个文件夹 |
| `src/pages/rpo/index.tsx` | RPO及猎头服务的完整页面，组装各个组件 |
| `src/components/layout/` | 全站通用的布局组件，如导航栏、页脚 |
| `src/components/rpo/` | RPO页面专用的各个区块组件 |
| `src/types/rpo.ts` | RPO页面用到的 TypeScript 类型定义 |
| `src/App.tsx` | 应用根组件，配置页面路由 |

## 数据处理方式

⚠️ 本页面为纯静态页面，所有数据直接写在代码中。

推荐方式：在组件文件顶部定义常量数组/对象，例如：

```typescript
const SERVICE_STEPS = [
  {
    id: 1,
    phase: "前期",
    title: "百得思维RPO服务简介",
    description: "向客户初步介绍百得思维的核心招聘行业与领域..."
  },
  // ...
] as const;
```

类型定义放在 `src/types/rpo.ts` 中，用于约束数据结构的一致性。

## 附录：通用编码准则

以下是通用的 LLM 编码行为准则，用于减少常见错误。

### 1. 思考再编码

**不假设、不隐藏困惑、明确权衡。**

实施前：
- 明确说明假设。不确定时就问。
- 如果有多种解释，列出它们——不要默默选择。
- 如果有更简单的方法，说出来。必要时提出反对意见。
- 如果有不清楚的地方，停下来。指出困惑点。提问。

### 2. 简洁优先

**用最少的代码解决问题。不做推测性开发。**

- 不添加未被要求的功能。
- 单次使用的代码不做抽象。
- 不添加未被要求的"灵活性"或"可配置性"。
- 不为不可能的场景添加错误处理。
- 如果写了 200 行但其实 50 行就够，重写。

问自己："资深工程师会认为这过于复杂吗？"如果是，简化。

### 3. 精准修改

**只改必须改的。只清理自己造成的混乱。**

编辑现有代码时：
- 不"改进"相邻的代码、注释或格式。
- 不重构没坏的东西。
- 匹配现有风格，即使你会用不同方式。
- 如果注意到无关的死代码，提一下——不要删除。

当你的改动产生孤立代码时：
- 移除**你的改动**导致未使用的 imports/变量/函数。
- 不要移除预先存在的死代码，除非被要求。

测试标准：每一行改动都应该直接追溯到用户的请求。

### 4. 目标驱动执行

**定义成功标准。循环验证直到达成。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入写测试，然后让测试通过"
- "修复 bug" → "写一个能重现它的测试，然后让测试通过"
- "重构 X" → "确保重构前后测试都通过"

对于多步骤任务，陈述简要计划：
```
1. [步骤] → 验证：[检查]
2. [步骤] → 验证：[检查]
3. [步骤] → 验证：[检查]
```

强成功标准让你能独立循环。弱标准（"让它工作"）需要持续澄清。

---

**这些准则有效的标志：** diff 中不必要的改动更少，因过度复杂导致的重写更少，澄清性问题出现在实现之前而不是犯错之后。
